# Bonus：Human-Agent Iteration Architecture / Validation & Observability

## 1. Human-Agent Iteration Architecture

這裡指的不是程式碼的系統架構，而是「人類與 Agent 之間如何來回迭代」這件
事本身的架構設計。本專案實際發生的迭代迴圈如下：

```
┌─────────────┐     2 個選擇題       ┌──────────────┐
│   使用者    │ ───────────────────▶ │   Agent      │
│ (給出題目)  │ ◀─────────────────── │ (澄清範圍)   │
└─────────────┘     範圍 + 技術棧     └──────┬───────┘
                                              │
                                              ▼
                                     ┌──────────────────┐
                                     │ 規劃層             │
                                     │ ARCHITECTURE.md   │ ← 人類可在此插入
                                     │ (決策先於程式碼)   │   review/糾正點
                                     └────────┬──────────┘
                                              │
                                              ▼
                                     ┌──────────────────┐
                                     │ 實作層             │
                                     │ data → composable  │
                                     │ → store → 元件      │
                                     │ → 頁面              │
                                     └────────┬──────────┘
                                              │
                                              ▼
                                     ┌──────────────────┐
                                     │ Agent 自我驗證層    │ ← 人類完全不需介入
                                     │ build/typecheck/    │   的內部迴圈
                                     │ curl 實測 HTML       │
                                     └────────┬──────────┘
                                              │
                                              ▼
                                     ┌──────────────────┐
                                     │ 交付 + 協作分析報告 │ ← 人類在此重新進入
                                     └──────────────────┘   迴圈（驗收/下一輪需求）
```

這個架構的核心設計理念是**把迭代迴圈分成兩種不同回饋速度的層**：

- **快迴圈（Agent 內部）**：build 錯誤、型別錯誤、執行期錯誤。這類問題
  不需要、也不應該丟給人類來迭代——讓人類看一次 TypeScript 報錯再回來
  說「這裡有 bug」是在浪費人類的回合數。本專案中兩個錯誤（缺漏 import、
  型別檢查失敗）都在這層被攔截、修正、重新驗證，全程沒有消耗人類的
  任何一次來回。
- **慢迴圈（人機之間）**：範疇、技術棧、視覺風格、取捨決策這類「正確答案
  依賴使用者意圖」的問題，才值得佔用人類的回合數。本專案把這類問題壓縮
  成開頭 2 個選擇題，把之後所有「能自己決定」的細節都收斂進 Agent 自主
  決策，並把決策記錄寫成文件，讓人類事後審查時有完整脈絡可以追溯「為什麼
  是這樣」，而不需要重新解釋一次。

這種分層的價值在於：**迭代次數不是越多越好，迴圈分配的層次才是關鍵**。
把該留給 Agent 自己解決的問題（編譯錯誤、型別問題）攔在快迴圈裡，把真正
需要人類判斷的問題（這個專案的範疇、這個視覺風格對不對）保留給慢迴圈，
才能讓「人機協作效率」這個指標有意義，而不只是迭代次數變少。

## 2. Validation Thinking（驗證思路）

本專案實際採用、且在上面章節有留下證據的驗證手段，依「驗證能攔住什麼層級
的錯誤」分層說明：

| 層級 | 手段 | 攔住的錯誤類型 | 本專案實際證據 |
|---|---|---|---|
| 靜態 | `nuxi typecheck` | 型別不符、可能為 undefined 的存取 | 攔到 `usePlaceholderImage.ts` 的陣列解構問題 |
| 建置期 | `nuxi build` | import 缺漏、語法錯誤、SSR 環境下無法執行的程式碼（如直接用 `window`） | 攔到 `Product` 型別缺漏 import（在 typecheck 前的編譯階段就會報錯，這裡實際是先被人工檢視抓到） |
| 執行期（黑箱） | 啟動真實 Node server，用 `curl` 對每條路由發送請求並檢查 HTTP 狀態碼 | 路由設定錯誤、伺服器端執行期例外（500）、404 處理失敗 | 驗證 `/goods/NOTEXIST` 正確回傳 404，其餘路由皆 200 |
| 執行期（內容層） | 對回應的 HTML 內容做字串層級斷言（如 grep 特定文字是否存在） | SSR 渲染策略名實不符（宣稱 SSR 卻沒渲染出內容；宣稱 CSR 卻不小心洩漏到 SSR） | 驗證 `/goods/M00001` 的 `<title>` 確實含有真實商品名稱；驗證 `/search` 的初始 HTML 含有「共 14 件商品」字樣；驗證 `/discover` 與 `/live` 的初始 HTML **不含**商品/直播卡片標記，證明 CSR-only 設定確實生效 |

最後一層（執行期內容層驗證）是最容易被省略、卻最關鍵的一層，因為前三層
只能告訴你「程式沒有壞」，無法告訴你「架構文件宣稱的渲染策略是不是真的
照做了」。本專案在這層做的具體驗證，正是把 `ARCHITECTURE.md` 第 3 節的
SSR/CSR 決策表，轉成可被 curl 重現的斷言——這代表這份文件不是寫完就放著
的「願景文件」，而是有對應的驗證手段去確認程式碼真的落實了文件裡的決策。

> **驗證過程中的真實插曲**：用 curl 測試 `/search?q=耳機` 時第一次得到
> 400，一度懷疑是中文查詢字串處理出了 bug。實際追查後發現是 shell 把
> 未經 URL encode 的原始 UTF-8 字元直接丟給 curl 造成的傳輸層問題，
> 改用 `--data-urlencode` 正確編碼後就回傳 200。這個插曲本身也是
> Validation Thinking 的一部分：**驗證工具本身的使用方式也可能是
> 雜訊來源**，在判定「這是程式的 bug」之前，要先排除「這是我測試方式
> 的 bug」，否則會把寫程式的力氣花在修一個其實不存在的問題上。

## 3. Observability Thinking（可觀測性思路）

純前端 Mock 專案沒有真實後端，傳統意義上的 APM / 日誌系統用不上，但
「可觀測性」這個概念仍然有務實的落地方式，本專案採用以下幾個設計：

1. **Mock API 刻意保留人工延遲（`delay()`）**：`useMockApi.ts` 裡每個
   函式都有 80–250ms 不等的模擬網路延遲。這不是真的在模擬網路狀況，而是
   讓 loading/skeleton 狀態在開發與展示時「有機會被看見」——如果資料
   瞬間回來，Skeleton 元件寫對寫錯根本看不出差異，等於做了一個永遠不會
   被驗證的功能。延遲本身就是一種低成本的可觀測性設計：讓本來會被忽略
   的中間狀態變成可觀察、可檢驗的東西。
2. **Rendering 策略表本身是一種架構可觀測性**：把每條路由的 SSR/CSR
   決定明文寫在 `nuxt.config.ts` 的 `routeRules` 裡，而不是讓它隱含在
   程式碼的某個角落（例如散落各頁面的 `<ClientOnly>` 標籤），代表任何人
   （包含未來的 Agent 自己）只要看設定檔就能一眼看出整站的渲染策略分佈，
   不需要逐頁檢查程式碼才能知道。
3. **誠實的空狀態與停用狀態，而非沉默失敗**：購物車「結帳」按鈕用
   `disabled` + 文字說明標示「未實作」，搜尋無結果有 `EmptyState`
   元件明確告知，而不是讓使用者面對一個空白畫面去猜「是壞了還是本來
   就沒東西」。這是把「系統內部狀態」主動暴露給使用者，而非要使用者
   自己回頭去問 Agent「這個功能是不是壞了」——某種意義上，這把原本該
   留給開發者除錯用的可觀測性，前移成了使用者也能直接看懂的介面語言。
4. **本報告與架構文件本身，就是這次協作過程的可觀測性紀錄**：記錄了
   Agent 在過程中產生的兩次錯誤與修正方式，而不是只呈現「最終結果一切
   正常」的乾淨敘事。這對於評估 Agent 協作品質是必要的——如果評估者
   只能看到結果，無法看到過程中哪些環節容易出錯、又是怎麼被攔下來的，
   就無法判斷這套協作流程在更大規模任務上是否可靠。
