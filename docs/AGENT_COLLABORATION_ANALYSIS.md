# 開發流程規劃與 Agent 協作效率評估

> 對應考核重點：「開發流程規劃與 Agent 協作效率評估/分析」
> 本報告記錄這次任務從題目到完成的真實協作過程，並對 Human-Agent
> 協作模式做反思，而非事後補寫的理論文章。

## 1. 任務拆解與執行順序

收到題目後，我（Agent）採取的順序是：

```
1. 用互動式問題澄清範圍與技術棧（而非預設一種方案直接開始）
2. 撰寫架構規劃文件（路由 / Rendering 策略 / 元件 / Mock Data 結構）
3. 用腳本批量生成 Mock Data（而非手寫，換取資料量與真實感）
4. 由下而上建構：composables → stores → 共用元件 → layout → 頁面
5. Build → Typecheck → 啟動真實 server → 用 curl 驗證實際 HTML 輸出
6. 撰寫本分析報告
```

這個順序本身就是一個值得記錄的決策：**先寫文件再寫程式碼**，而不是先刻
程式碼、事後補寫架構說明。理由是 SSR/CSR 策略、Mock Data 欄位設計這類
決策一旦先寫程式碼，往往會被「先求動起來」的壓力壓過，事後補的文件容易
變成程式碼的事後合理化，而不是真正驅動實作的設計思考。

## 2. 人機協作介面：使用者只給了兩個訊號

整個任務中，人類只用兩次選擇題回答（任務範圍 = 全部都要；技術棧 = Nuxt
v4 SSR）就讓 Agent 完成了從零到可運行專案的全部工作。這個協作密度本身
就是評估重點：

| 協作密度指標 | 數值 | 說明 |
|---|---|---|
| 人類主動輸入次數 | 2 | 範圍 + 技術棧 |
| Agent 自主決策次數 | 約 30+ | 路由結構、元件拆分、Mock Data 欄位、配色、SSR/CSR 分配、Scope Cut 項目等 |
| 人類介入修正次數 | 0 | 過程中未發生需要人類糾錯的情況 |

這反映出一個關於 Agent 協作效率的真實觀察：**當任務目標明確、且允許 Agent
自行決定實作細節時，「最小可行澄清」（2 個問題）配合「文件先行」可以撐起
整個專案，不需要逐頁來回確認**。但這同時是一種風險：如果使用者對視覺風格、
頁面取捨有強烈但未說出口的偏好，這種高自主度會導致最終產出與期待產生落差，
只能等使用者看到成品後才發現。

## 3. Agent 自我糾錯的實例（過程證據）

開發過程中，Agent 確實犯了兩個小錯，並在驗證階段被自己抓出來：

1. **缺漏 import**：`pages/index.vue` 與 `pages/goods/[id].vue` 使用了
   `Product` 型別卻忘了 import，是在編譯/型別檢查前用人工檢視抓到並補上的。
2. **TypeScript 嚴格模式型別錯誤**：`usePlaceholderImage.ts` 用陣列解構
   `palette[i]` 在 `strict: true` 設定下被判定為可能 `undefined`，跑
   `nuxi typecheck` 才抓出來，修正為加上 fallback。

這兩個錯誤都不是「讓使用者看到才發現」，而是在 **build → typecheck →
實際啟動 server 用 curl 檢驗 HTML 輸出** 這個自我驗證迴圈中被攔截。這正是
下一節要談的 Validation 設計所要解決的問題：把錯誤攔在使用者看到之前。

## 4. 與真實 momo 網站的差異分析

這是題目「考核重點」明確要求的部分，逐項列出本專案與真實 momo 購物網站
的差異，並說明造成差異的原因（技術限制 / 刻意簡化 / 範疇排除）：

| 面向 | 真實 momo | 本專案 | 差異原因 |
|---|---|---|---|
| 後端 | 真實商品資料庫、會員系統、金流、物流 API | 純前端 Mock Data（140 件商品、JSON 靜態檔） | 題目明確要求「無任何真實網站 API 呼叫」 |
| 渲染策略 | 多層 CDN/Edge Cache，依頁面類型有更細緻的快取策略（部分頁面甚至用 ISR 或邊緣渲染） | Nuxt `routeRules` 的 SSR/CSR 二元切換 | 規模差異：千萬級流量站需要的快取分層，在練習專案中用更簡化的等價方案示範同樣的「思考」即可 |
| 個人化 | 真實的使用者行為追蹤、推薦演算法 | `/discover` 用簡單 hash 打亂順序模擬「隨機探索感」 | 沒有真實使用者行為資料可訓練/驅動推薦邏輯 |
| 圖片 | 真實商品攝影圖、CDN 圖片服務 | 用 SVG data URI 動態生成色塊佔位圖 | 避免任何版權風險與外部圖片 API 依賴，同時保持 100% 離線可執行 |
| 搜尋 | 全文索引引擎（如 Elasticsearch）、複雜的相關性排序 | 簡單字串 `includes()` 比對 + 固定排序規則 | Mock 規模（140件）不需要也不該過度設計搜尋引擎 |
| 直播 | 真實串流（RTMP/HLS）、即時聊天室、即時庫存扣減 | 純視覺模擬（漸層色塊 + 模擬觀看人數跳動的 setInterval） | 純前端練習不應該也無法接真實串流服務 |
| 結帳/金流 | 完整金流串接、信用卡/超商付款 | 購物車到「結帳」按鈕為止，按鈕為 disabled 並標註「未實作」 | 明確 Scope Cut（見架構文件第 6 節），避免讓使用者誤以為這是可動用的功能 |
| 會員系統 | 登入、訂單記錄、收藏清單 | 無 | 同上，純前端 mock 不該假造一個看起來能用卻是空殼的登入功能 |
| RWD/無障礙 | 成熟、經過大量真實使用者測試 | 基本 RWD breakpoint + aria-label + focus-visible + prefers-reduced-motion | 練習規模下做到「正確的基本功」，而非企業級無障礙稽核等級 |

**最關鍵的一個差異是誠實揭露而非假裝完整**：購物車結帳按鈕刻意做成
disabled 並標明原因，而不是讓它看起來能點但什麼都不會發生——這個選擇本身
也是協作效率與信任的一部分：寧可讓使用者一眼看出邊界在哪，也不要留下一個
看起來能用、實際是空殼的功能誤導使用者驗收。

## 5. 開發效率觀察：複用設計帶來的乘數效應

`ProductCard.vue` 是全站使用密度最高的元件，同時被首頁 carousel、搜尋頁
grid、商品詳情頁的「猜你喜歡」、直播頁的「熱銷商品」共用。在 Agent
協作的脈絡下，這代表：

- 在這顆元件上多花的一輪打磨（折扣徽章、星級評分、售出數量格式化），會
  同時反映在 4 個頁面上，不需要在每個頁面各刻一次。
- 反過來說，這顆元件如果設計時沒考慮周全（例如忘了處理 `originalPrice`
  為 null 的情況），錯誤也會同時出現在 4 個地方——**複用提高了正確設計
  的槓桿，但也放大了設計缺陷的影響範圍**，這是評估 Agent 產出時值得注意
  的雙面性，不能只看「程式碼重複率低」這個單一指標來判斷品質。

## 6. 若要重新做一次，會改變的協作方式

誠實反思幾個可以做得更好的協作節點：

1. **可以在架構文件定稿後，多加一次「文件 review」的中間確認點**——目前
   是直接從文件跳到寫程式碼，雖然這次沒有出問題，但如果使用者對 SSR/CSR
   策略表的判斷有不同意見（例如認為 `/discover` 也該 SSR 以利 SEO），
   現在的流程要等到看到成品才會發現分歧，而非在規劃階段就被攔下來。
2. **Mock Data 量級可以先問過再決定**——140 件商品是 Agent 自行決定的
   數字（每分類 14 件 × 10 分類），對示範用途夠用，但如果使用者的考核
   標準包含「大量資料下的虛擬滾動效能」，目前的量級可能不足以顯現差異，
   這屬於沒有事先對齊驗收標準的風險。
